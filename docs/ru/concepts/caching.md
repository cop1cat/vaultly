# Кэширование

Каждая `SecretModel` имеет thread-safe TTL-кэш. Смысл vaultly в том,
что вы про него не думаете — но если думаете, то вот как он устроен.

## Один кэш на корень

Каждый корневой инстанс `SecretModel` владеет одним `TTLCache`.
Вложенные дети делят кэш своего корня (и его бэкенд). Два несвязанных
корня имеют независимые кэши; ротация одного не инвалидирует другой.

```python
prod = App(stage="prod", backend=b)
dev  = App(stage="dev",  backend=b)

prod.refresh_all()  # очищает только кэш prod; dev не тронут
```

## Ключи

Ключ кэша — это **резолвленный путь** (с подставленной `{var}`),
опционально с суффиксом `@version` для пин-версий:

```text
/prod/db/password         # не-версионированный секрет
/prod/db/password@2       # версия 2 того же секрета
```

Два разных поля, дающих один и тот же ключ, делят одну запись в кэше —
и один фетч из бэкенда.

## Семантика TTL

Задаётся per-field через `ttl=` в `Secret(...)`:

| Значение         | Поведение                                                   |
| ---------------- | ----------------------------------------------------------- |
| `None` (default) | Запись не истекает.                                         |
| `0`              | Запись сразу stale; каждое чтение идёт в бэкенд.            |
| `> 0`            | Запись живёт `ttl` секунд.                                  |

Просроченные записи не удаляются при чтении — они лежат, пока их не
перезапишет новый фетч или `invalidate` / `clear`. Это то, что читает
`stale_on_error`.

## Поток `prefetch()`

`prefetch()` (также триггерится через `validate="fetch"`) обходит всё
дерево моделей, делит секреты на версионированные и не-версионированные,
и:

1. **Не-версионированные**: один вызов `backend.get_batch(unique_paths)`.
   Per-key locks захватываются *до* batch-вызова, чтобы конкурентный
   читатель не проскочил и не сделал двойной фетч.
2. **Версионированные**: серия `backend.get(path, version=...)` (ни один
   batch API не поддерживает per-path версии).

Каждое значение кастуется в тип своего поля и кладётся под свой ключ
кэша. Идемпотентен — повторный вызов `prefetch()` при тёплом кэше
дешёвый (но всё равно идёт в бэкенд; для «только тёплый кэш»
используйте дефолтный ленивый режим).

## Модель конкурентности

vaultly сериализует только то, что необходимо:

- **Hot-чтения** (cache hit) берут только внутренний lock кэша — не
  per-key. Они масштабируются по потокам.
- **Cold-чтения** (промах для конкретного ключа) берут per-key lock,
  делают повторную проверку кэша, потом фетч. Несколько потоков,
  гонящихся за *одним* холодным ключом, видят ровно один вызов бэкенда.
- Потоки, гонящиеся за *разными* холодными ключами, друг другу не мешают.
- `refresh(name)` держит per-key lock на `invalidate + fetch`.
  Конкурентный читатель либо получит старое значение (кратко), либо
  дождётся нового фетча.

## Инвалидация

| API                    | Эффект                                              |
| ---------------------- | --------------------------------------------------- |
| `model.refresh(name)`  | Удалить один секрет из кэша и перефетчить.          |
| `model.refresh_all()`  | Удалить весь кэш.                                   |
| (TTL expiry)           | Чтение поднимает «miss» внутри; идёт фетч в бэкенд. |

Заметьте, что `refresh_all()` чистит **весь кэш**, включая поля с
`ttl=None`. После события ротации это обычно нужный молоток.

## Жизненный цикл `_fetch_locks`

`_fetch_locks` — это `KeyedLocks` (один `RLock` на резолвленный путь).
Locks накапливаются по мере фетча новых ключей. Для большинства
приложений набор путей ограничен формой модели и это нормально.

Multi-tenant приложения, использующие `tenant_id` в путях (и видящие
тысячи уникальных путей со временем), могут вызвать
`model._fetch_locks.discard(key)` или `model._fetch_locks.clear()` на
подходящих границах для освобождения памяти. Аналогично с `_cache`:
`model._cache.clear()`.
